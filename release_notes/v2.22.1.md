# v2.22.1

## 🎯 Highlights

Focused maintenance patch on the v2.22 line. Two `sunset_position` bugs are fixed: covers sitting at the sunset/night position all morning when `start_time` precedes astronomical sunrise ([#438]), and a config-flow data-persistence bug that made it impossible to clear a previously saved `sunset_position = 0` ([#439]). No new features, no breaking changes.

**This is NOT the stable release of the v2.22.1-beta.X series.** See the section below.

---

## 🔁 About this release line

The v2.22.1-beta.1 through v2.22.1-beta.4 tags were cut from `main` as part of the v2.23 development cycle. They share only a version-number prefix with this release — the codebase diverged from v2.22.0 in a different direction. If you installed a v2.22.1-beta.X build, you are on the v2.23 pre-release track and **will not be upgraded to this stable**. Stay on that track to receive v2.23 when it ships, or downgrade to v2.22.0 and install v2.22.1 to stay on the v2.22 maintenance line.

v2.22.1 was built from the `release/v2.22` maintenance branch and contains exactly one commit on top of v2.22.0: the backport of PR [#451], which cherry-picks the two fixes below.

## 🐛 Fixes

- **`sunset_position` applied during the morning operational window when `start_time` came before astronomical sunrise ([#438]).**

  On short winter days where the user's configured `start_time` (e.g. 08:00) falls before astronomical sunrise (e.g. 08:15), `compute_effective_default()` in `helpers.py` evaluated the `before_sunrise` branch as true even though the operational window had already opened. Covers sat at `sunset_position` instead of computing a solar position for the full duration between `start_time` and actual sunrise.

  The fix adds an `after_start_time` keyword argument to `compute_effective_default`. When `True`, the `before_sunrise` condition is suppressed. The coordinator passes `self.after_start_time` at both call sites, so the night-position rule no longer fires once the user's window is open.

- **`sunset_position = 0` could not be cleared in the config flow ([#439]).**

  `POSITION_SCHEMA` treats `sunset_position` as an optional field with `vol.UNDEFINED` as its default. When a user submitted the position step with the field blank (intending to clear it), `POSITION_SCHEMA.merge(prior_options)` re-applied the stored `0` from prior options — since `0` is falsy, it survived the merge and the clear never took effect.

  The fix introduces `_POSITION_OPTIONAL_KEYS` — a list containing `CONF_SUNSET_POS`, `CONF_MY_POSITION_VALUE`, `CONF_SUNSET_TIME_ENTITY`, and `CONF_SUNRISE_TIME_ENTITY` — and calls `self.optional_entities(_POSITION_OPTIONAL_KEYS, user_input)` in both `ConfigFlowHandler.async_step_position` and `OptionsFlowHandler.async_step_position` before the dict update. Omitted fields are removed from stored options rather than inheriting the prior value.
