# v2.22.1-beta.2

beta.2 fixes three bugs found since beta.1: the in-transit guard that protects against false manual-override events was not applied during the reconciliation pass, the custom-position decision-trace attributes introduced in beta.1 were renamed and the pipeline registry bug that prevented them from reaching the sensor was corrected, and the `cover_type` attribute was missing from `sensor.<cover>_control_status` (causing awning wedge-fill to render backwards in the companion Lovelace card).

## ­¤Ä» Should I install this?

- **If you use the companion Lovelace card with awning covers**, install this ŌĆö the wedge-fill polarity fix requires `cover_type` to be present in the `control_status` sensor attributes.
- **If you are waiting for the beta.1 custom-position trace attributes**, the field names changed in beta.2: `active_slot` ŌåÆ `custom_position_active_slot` and `floor_binding` ŌåÆ `custom_position_minimum_mode`. Update any templates or dashboard cards before upgrading.
- **If you have slow or RF-controlled covers** that have ever unexpectedly snapped back to an ACP-managed position, the reconciliation in-transit fix (#418) removes a latent race that could trigger a re-send while the cover was mid-travel.
- **If you are on v2.22.0 and it is working**, all three fixes are low-risk. No existing HA state or attributes change beyond the items above.

## Ō£© Features

- **`custom_position_active_slot` and `custom_position_minimum_mode` on the decision trace ([#421], PR [#424]):**

  The `active_slot` and `floor_binding` fields introduced in beta.1 have been renamed to avoid collision with other attribute namespaces and to better describe their semantics:

  **`custom_position_active_slot`** ŌĆö the 1-based slot number (1ŌĆō4) of the winning Custom Position handler. `None` when any other handler wins.

  **`custom_position_minimum_mode`** ŌĆö present only when `custom_position_active_slot` is set and `min_mode` is active. `True` when the configured floor is raising the cover above the raw autonomous calculation (the floor is actively constraining). `False` when the raw calculation already meets or exceeds the floor (the floor is a no-op). `None` in exact mode (`min_mode=False`) or when any non-custom handler wins.

  The same PR also fixes the pipeline registry bug that prevented these fields from propagating through to the sensor: `PipelineRegistry.evaluate` now uses `dataclasses.replace` to build the `PipelineResult` from the winning handler, so all fields on the winner ŌĆö including the two custom-position diagnostics ŌĆö carry through without being enumerated individually. `sensor.py`'s `_decision_trace_attrs` function exposes the renamed fields.

## ­¤Éø Bug Fixes

- **Reconciliation pass honors in-transit guard ([#418], PR [#423]):** The `handle_state_change` path in `AdaptiveCoverManager` has rejected mid-travel position events since issue #271, preventing false manual-override detection when HA's reported position lags physical position during a transit move. The reconciliation pass in `CoverCommandService` lacked an equivalent check: a reconcile tick landing while a cover was mid-travel could re-send `set_cover_position` toward the prior ACP target and override a user's RF command. The fix extracts `_is_cover_in_transit` on `CoverCommandService` (reading `hass.states.get(entity_id).state in ("opening", "closing")`) as a shared predicate. The reconciliation loop skips any in-transit entity and records a `reconcile_skipped_in_transit` event; `handle_state_change` delegates to the same predicate via the new `is_in_transit` callback parameter on `AdaptiveCoverManager.handle_entity_state_change`.

- **`cover_type` now present on `control_status` sensor attributes:** `sensor.<cover>_control_status` was missing the `cover_type` attribute. The companion Lovelace card reads this attribute to determine wedge-fill polarity (awnings: extended = full, retracted = empty; blinds: closed = full, open = empty). Without it, every cover fell back to `cover_blind` behavior, so a fully-extended awning rendered as an empty wedge. The fix adds `"cover_type": s._cover_type` to `_control_status_attrs` in `sensor.py`.
