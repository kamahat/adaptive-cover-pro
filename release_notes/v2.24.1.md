A focused patch release fixing four cover-control regressions and tidying up the options menu with a small acknowledgement link.

This patch addresses a tilt-only venetian custom-position regression, a night-hold failure when the schedule start time is left blank, a cloud-suppression handler visibility gap in diagnostics, and a floor-trace labelling mismatch with sparse custom-position slot configurations. A Buy Me a Coffee link has also been added to the options menu, and the URL was relocated to satisfy HA hassfest validation.

---

## 🚀 Features

### Buy Me a Coffee link in the options menu

A small acknowledgement link now surfaces on the integration's options menu so users who find the integration useful have an easy way to show their appreciation.

---

## 🐛 Fixes

### Custom position now honoured on tilt-only venetian configurations ([#499], [#502])

On tilt-only venetian setups, custom-position sensors were being silently ignored: the cover remained at its calculated tilt instead of snapping to the user-pinned position. Any sensor configured with an explicit user intent — force override, custom position, and similar methods — was treated the same as an auto-calculated result.

`VenetianPolicy` now consults the `_EXPLICIT_USER_POSITION_METHODS` frozenset, which includes `ControlMethod.CUSTOM_POSITION`, `ControlMethod.FORCE`, and the other explicit-intent control methods. When the active control method is a member of that set, the user-pinned position is applied unconditionally, regardless of tilt-only geometry.

### Night hold restored when schedule start time is blank ([#492], [#493])

When the schedule's start time field was left blank, the integration mishandled the "no start configured" case and stopped holding the night position after midnight on setups that configured only an end time. The cover would move unexpectedly instead of waiting for the end-time boundary.

`TimeWindowManager` now recognises the `BLANK_TIME` sentinel (`"00:00:00"`) explicitly. The `_start_has_passed()` method and the `window_explicitly_started` property together gate `compute_effective_default()`, keeping the night position locked until the configured end time elapses.

### Cloud-suppression handler now activates for every light and cloud sensor source ([#494], [#495])

When users configured a lux sensor, irradiance sensor, or is-sunny binary sensor — but not the cloud-coverage entity — the cloud-suppression handler did not appear in the diagnostics' enabled-handlers list. The handler ran silently, making it invisible when troubleshooting.

The diagnostics surface now checks each of `CONF_CLOUD_COVERAGE_ENTITY`, `CONF_IRRADIANCE_ENTITY`, `CONF_IS_SUNNY_SENSOR`, and `CONF_LUX_ENTITY` independently; any one being configured is sufficient to mark the handler as enabled.

### Custom-position floor trace now labels each step with its real slot number ([#496], [#497])

With gaps between the four custom-position floor slots — for example slots 1 and 3 configured, 2 and 4 empty — the diagnostics decision trace labelled the second active row as `custom_position_2` instead of `custom_position_3`. The mislabelling occurred because the name was synthesised from the iteration index of the active list rather than the original slot number.

`custom_position_handler_name()` is now the single source of truth for producing the canonical handler name. It accepts a `CustomPositionSensorState.slot` value and emits the correct name; every site that previously constructed the name from the list index now calls this helper instead.

### Buy Me a Coffee URL relocated for HA hassfest compliance

HA's hassfest validation flags hard-coded URLs embedded in field labels. The Buy Me a Coffee URL has been moved into `description_placeholders` under the `{coffee_url}` token so the label text is plain and the URL is injected separately, satisfying the compliance check without changing the user-visible result.
