## Highlights

v2.30.2 adds daytime gate resilience alongside four bug fixes and a French translation correction. The headline change stops transient sensor outages from triggering spurious position changes: when all gate sensors go unavailable, the gate holds its last-known verdict for two minutes before falling back to astral. The fixes address a diagnostics lookup that broke after entry reload, endpoint enforcement silently ignoring 0 and 100 as special positions, same-position tolerance applying too broadly at mid-range positions, and 13 phrasing errors in the French UI.

## Added

- **Daytime gate holds last-known verdict and falls back to astral (#742, #745):** A new `GracefulSource` class tracks a tri-state gate verdict (daytime / dark / indeterminate) and, when all configured sensors go unavailable, holds the last-known result for up to `DEFAULT_DAYTIME_GATE_GRACE_SECONDS` (120 s) before switching to astral sunrise/sunset. Sensors that recover within the grace window restore sensor-driven operation immediately, with no astral fallback ever triggered. Sensors that stay unavailable past the window get a clean handoff to astral rather than an abrupt flip.

## Fixed

- **Diagnostics coordinator resolved from `entry.runtime_data` (#753, #754):** The diagnostics export previously looked up the coordinator through the legacy `hass.data[DOMAIN]` path, which fails during an entry reload. It now resolves through `entry.runtime_data`. A reload-window cache keyed by `DIAG_CACHE_KEY` — written by `_cache_last_good_diagnostics()` — lets `async_get_config_entry_diagnostics()` serve the last-good snapshot while the coordinator is temporarily absent.

- **Endpoint enforcement now drops 0 and 100 from the special-positions list (#679, #752):** When endpoint enforcement is on, positions 0 and 100 must be sent unconditionally. Previously, if a sunset or default position was configured to 0 or 100, that value entered the special list and the delta gate skipped the command. `filter_endpoint_specials()` now strips 0 and 100 from that list when enforcement is active; `check_position_delta()` and `build_special_positions()` both use it.

- **Same-position tolerance scoped to endpoints 0 and 100 only (#507, #746):** The `_at_target()` helper applies position tolerance only when the target is 0 or 100; all other positions require exact equality. `apply_position()`, `check_target_reached()`, and `run_reconciliation_pass()` all call `_at_target()` consistently. Previously the tolerance band applied uniformly, causing mid-range special positions to be silently swallowed when the cover had drifted by a small amount.

- **French translation corrections (#738, #743):** 13 phrasing errors in `fr.json` corrected, covering terminology ("couverture" → "store") and standard phrases ("En savoir plus" for "Learn more"). No schema changes.

## Testing

5,321 tests passing. The venetian test suite was refactored to share a single `neutralize_venetian_delays` fixture, replacing three duplicated per-file fixtures and cutting serial runtime from 58 s to 2 s for those files. A new `./scripts/test fast` command runs the suite in parallel without `pytest.mark.perf` wall-clock budget tests.

Requires Home Assistant 2026.3.0+. The companion Lovelace card ([jrhubott/adaptive-cover-pro-card](https://github.com/jrhubott/adaptive-cover-pro-card)) is a separate repo with its own release cycle.
